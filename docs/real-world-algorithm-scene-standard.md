# 每个算法真实场景开发标准

本文档是后续开发每个算法 3D 真实场景的统一标准。新增、重构或评审算法场景时，以本文档为准。

## 1. 核心目标

每个算法场景都必须像一个真实世界里的功能区，而不是抽象演示台。

算法数据要嵌入真实物体，算法步骤要驱动真实任务动画，用户运行算法后应能直观看到“现实任务如何被算法推进”。

## 2. 场景定位要求

每个算法必须先定义一个具体真实场景。

示例：

| 算法类型 | 推荐真实场景 | 数据含义 |
| --- | --- | --- |
| Dijkstra | 急救调度 / 城市导航 / 应急车辆派遣 | 建筑=节点，道路=边，cost=时间/距离/通行代价 |
| Prim / Kruskal | 城市微电网 / 管线铺设 / 通信网络建设 | 设施=节点，线缆/管道=边，权重=铺设成本 |
| Union-Find | 社区门禁合并 / 小区联网 / 道闸连通 | 建筑=元素，社区=集合，父节点=归属关系 |
| 排序 | 物流分拣中心 / 邮件仓库 / 包裹流水线 | 包裹=数组元素，位置=索引，数值=优先级/重量 |
| 搜索 | 图书馆 / 档案库 / 货架检索 | 物品=元素，指针=检索位置，目标=待查对象 |
| 树 | 苗圃 / 组织架构楼 / 文件档案树 | 节点=树对象，连线=父子关系，根=入口 |
| DP | 物流规划中心 / 生产排程 / 路线成本表 | 表格=状态，格子=子问题，箭头=状态转移 |
| 字符串 | 广告牌校验 / 文档匹配 / 印刷排版台 | 字符=文本块，窗口=匹配范围，指针=比较位置 |
| BFS / DFS | 消防搜救 / 安检巡逻 / 楼层搜索 | 节点=区域，边=通道，队列/栈=行动策略 |

禁止所有算法都使用同一种结构，例如“圆形场地 + 台子 + 小车 + 线 + 灯”。

## 3. 数据到现实物体的映射

每个数据结构必须映射到真实物体。

| 数据概念 | 必须映射为 |
| --- | --- |
| 节点 | 建筑、站点、设备、房屋、货架、库位、区域 |
| 边 | 道路、电缆、管线、传送带、走廊、桥、通道 |
| 数组元素 | 包裹、货箱、书本、卡片、设备槽位 |
| 索引 | 地面编号、货架编号、工位编号、门牌号 |
| 权重 / cost | 路程、时间、费用、阻力、施工成本 |
| 指针 | 工人、机械臂、扫描器、巡逻员、车辆、光标 |
| 集合 / parent | 社区归属、门禁系统、楼栋管理关系 |
| DP 状态 | 控制台格子、规划表格、任务卡片、调度单元 |

要求：

- 真实物体必须和数据一一对应。
- 标注必须贴近对应物体，不能全部堆在空中。
- 当前步骤涉及的物体必须有明确状态变化。
- 必须能根据数据的更改动态构建场景。

## 4. 算法步骤动画要求

算法 trace 必须驱动场景动画，不能只改变灯光。

常见操作映射：

| 算法操作 | 推荐动画 |
| --- | --- |
| visit / 访问 | 车辆到达、人员进入、设备启动、区域亮起 |
| relax / 松弛 | 车辆沿道路行驶，候选道路高亮，cost 更新 |
| compare / 比较 | 两个包裹被扫描器扫过，机械臂对准 |
| swap / 交换 | 两个真实物体换位，路径和目标位置可见 |
| union / 合并 | 两个社区道闸连通，门禁线合并 |
| find / 路径压缩 | 查询光线沿父节点回溯，再更新指向 |
| enqueue / dequeue | 人员/车辆进入或离开队列区域 |
| push / pop | 物体进入或离开栈式暂存架 |
| DP transition | 依赖箭头从已知状态流向当前格子 |

要求：

- 动画速度必须和播放速度同步。
- 动画路径必须和画出的道路/线缆/传送带一致。
- 如果视觉路径是折线，移动物体也必须沿折线移动，不能走直线。
- 当前步骤、上一步结果、候选对象要能区分。

## 5. 场景入口和相机要求

每个算法运行后必须直接跳到对应真实场景的正门口。

入口视角要求：

- 看得到场景主体。
- 看得到当前算法路径或当前操作区域。
- 看得到关键数据标注。
- 不能对着背墙、空白墙面、场景背面或地面角落。
- 相机复原力度要克制，避免用户手动调整视角后被强行拉回。

每个算法应在相机配置中有自己的入口位置和 target。

## 6. 视觉风格要求

每个算法场景必须有独立风格。

要求：

- 建筑布局要服务真实用途，不要所有场景都围成同一种圆形。
- 建筑颜色、材质、尺度、道具要符合场景职业属性。
- 真实环境可以有道路、行人、车辆、河流、云、太阳、灯光等，但不能抢算法主线。
- 天空和环境逻辑要合理。例如太阳和月亮不能同时作为主视觉出现。
- 河流、湖泊等自然元素必须有真实边界，不能像透明玻璃片盖在地面上，也不能淹建筑底部。

禁止：

- 数据路径和建筑入口不连接。
- 建筑过密导致路径和标注看不清。
- 大面积透明平面覆盖场景。
- 所有场景只靠亮灯表达算法。
- 只做卡通道具，不考虑现实功能。

## 7. 标注要求

每个场景必须有可读数据标注。

标注内容至少包括：

- 数据 id，例如 `node 0`、`index 3`、`parent -> 2`。
- 现实角色，例如 `急救中心`、`变电站`、`分拣口`、`档案架`。
- 当前状态，例如 `current`、`from/u`、`to/v`、`visited`、`settled`、`swap`、`compare`。
- 数值含义，例如 `cost 4`、`value 37`、`dist 6`、`rank 2`。

标注形式：

- 建筑类节点：入口地面牌 + 高处信息牌。
- 边 / 路径：沿道路放置 cost 牌和方向箭头。
- 数组元素：物体正面或顶部显示 value/index。
- 集合关系：房屋或社区门牌显示 parent/root。
- 当前操作：使用更高对比度，但不能遮挡主体。

## 8. 全屏说明按钮要求

每个算法真实场景都必须提供一个放大 / 说明按钮，用于展示全屏文字详细说明。

实现要求：

- 每个算法场景内必须有一个明显但不干扰画面的按钮。
- 按钮可参考旧代码中的 `showHelp` / `setShowHelp` 机制。
- 点击按钮后，打开全屏覆盖层，展示该算法场景的详细说明。
- 说明层必须是屏幕空间 UI，不应绑定在 3D 世界坐标里跟着相机乱动。
- 说明层必须支持关闭。

说明内容至少包括：

1. 场景名称。
2. 现实任务解释。
3. 数据映射表：节点/边/元素/权重分别对应什么。
4. 当前动画如何对应算法步骤。
5. 颜色和标注含义。
6. 用户应该如何观察该场景。

示例结构：

```tsx
const showHelp = usePlaybackStore(s => s.showHelp);
const setShowHelp = usePlaybackStore(s => s.setShowHelp);

// 3D 场景内按钮
<Html position={[x, y, z]} transform distanceFactor={8}>
  <button onClick={() => setShowHelp(true)}>?</button>
</Html>

// SceneFrame 或统一 overlay 中显示全屏说明
{showHelp && (
  <div className="fullscreen-help">
    <h2>Dijkstra 急救调度</h2>
    <p>建筑是图节点，道路是边，cost 表示通行时间。</p>
    <button onClick={() => setShowHelp(false)}>关闭</button>
  </div>
)}
```

后续建议将说明内容做成按算法 id 查询的配置，避免每个场景重复写 overlay。

## 9. 性能和懒加载要求

每个重场景必须考虑性能。

要求：

- 大型 3D 场景按需加载，不能全部塞进首屏主包。
- 重模型和贴图使用 `Suspense` 或动态 import。
- 重复物体优先使用 `memo`、实例化或轻量几何。
- 环境动画和算法动画分层，环境动画不能影响算法同步。
- 切换算法不能明显卡顿。
- 控制点光源数量，避免大量动态阴影。

当前标准：

- 城市主场景应作为独立 chunk 懒加载。
- GLB 车辆等资源应通过 `Suspense` 加载。
- 大量行人、树、灯、窗户优先使用轻量 mesh 或 instanced mesh。

## 10. 验收清单

新增或重构任何算法真实场景时，必须逐项检查：

- [ ] 这个算法有明确真实职业场景。
- [ ] 数据结构元素都映射到了真实物体。
- [ ] 算法步骤会驱动物体运动或状态变化。
- [ ] 当前操作不是只靠亮灯表达。
- [ ] 路径、线缆、传送带和移动动画完全一致。
- [ ] 建筑入口、道路、路径真实连接。
- [ ] 运行算法后相机跳到正门口，不看背面或空白墙。
- [ ] 场景有自己的颜色、建筑、材质和道具风格。
- [ ] 数据标注清楚，且不遮挡主体。
- [ ] 有全屏说明按钮。
- [ ] 全屏说明包含场景任务、数据映射、动画含义和颜色说明。
- [ ] 环境真实元素不抢算法主线。
- [ ] 没有太阳/月亮等环境逻辑冲突。
- [ ] 水体、地形等不会覆盖或淹没建筑。
- [ ] 场景懒加载或资源懒加载已考虑。
- [ ] 构建通过，并做过至少一次截图检查。
- [ ] 新建的场景不与原场地空间冲突。

## 11. 最终原则

如果真实感和算法可读性冲突，优先保证算法可读性。

如果装饰元素和算法动画冲突，删除或弱化装饰元素。

如果一个场景看起来像抽象演示台，而不是现实功能区，则不符合本文档标准。
